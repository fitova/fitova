# Homepage Refactor Plan — FITOVA v4

> **Authors:** Senior Product Architect, UI System Designer, Code Auditor  
> **Date:** 2026-03-03  
> **Status:** 🟢 IN PROGRESS — Phases 1–7 implemented (mobile device testing + final cleanup pending)  
> **Stack:** Next.js 14 App Router · TypeScript (strict) · Tailwind CSS · Supabase · Redux · TanStack Query · Framer Motion

## Executive Summary

FITOVA's homepage is a strong foundation but carries several accumulated technical debts and UX inconsistencies that diminish its premium affiliate platform identity. This refactor targets nine areas with surgical precision:

1. Hero section copy cleanup (remove "Wear Your Identity" phrase)
2. Categories section grid alignment fix (3-card layout misalignment)
3. Replace the generic ecommerce perks bar with affiliate-first value propositions
4. Product section mobile scroll experience (New Arrivals, Best Sellers, Trending)
5. Unified hover system across all product cards
6. Wishlist icon logic standardization with backend sync
7. Product navigation: replace modal with full product details page
8. Remove "OurView / ThisWeek" tabbed section entirely
9. 3D Section background gradient upgrade
10. Lookbook Preview section visual upgrade
11. Newsletter section redesign (away from blue/generic)

All changes must follow:
- Mobile-First strategy (320px → 2560px)
- Frontend architectural rules: Server Components default, TanStack Query for data, `/services` layer
- Backend architectural rules: RESTful versioned API (`/api/v1`), Zod validation, clean service layer
- Single Source of Truth: no duplicated logic, no hardcoded data floating in components

## Layout Fix Plan

### Fix 1: Hero Section — "Wear Your Identity" Removal

**Finding:** The phrase "Wear Your Identity" is not in static component code — it resides in `homepage_slides` Supabase records as slide subtitle or title content.

**Plan:**
- **Step A:** Update Supabase `homepage_slides` table via admin panel: remove or replace "Wear Your Identity" from any `title`, `subtitle`, or `description` field.
- **Step B:** Audit `HeroCarousel.tsx` fallback value at L54 (`"Redefine Your\nSignature Look"`) — this is clean, no change needed.
- **Step C:** After removing the phrase, verify layout: the `subtitle` span (L82–L84) feeds from `slide.subtitle`. Conditionally render the subtitle span only if `slide.subtitle` is truthy.
- **Step D:** Typography hierarchy after removal — H1 (`slide.title`) becomes more visually dominant. Verify `text-4xl sm:text-5xl lg:text-5xl` is sufficient. No structural changes needed if subtitle is conditionally rendered.
- **Step E:** No backend schema changes required — `homepage_slides` columns remain unchanged.

**Spacing judgment:** Removing subtitle creates ~`mb-6` (~24px) reclaimed whitespace above H1. This is acceptable — it makes H1 more prominent. Do NOT add padding compensation. Let whitespace breathe.

### Fix 2: Hero Section — Right Column Card Height Alignment

**Problem statement:** Left column = `HeroCarousel` with dynamic height (content-driven). Right column = 2 `Link` cards stacked vertically. The two right cards need to collectively match the left carousel height, with each card taking ~50% of that height.

**Current approach:** Both cards use `min-h-[150px]` — no max-height, no explicit 50% height.

**Plan — CSS Grid approach (no hardcoded heights):**

```
Outer wrapper: CSS Grid with grid-cols-[757fr_393fr] at xl
Left: single grid cell spanning full row height
Right: inner grid with grid-rows-[1fr_1fr] — each card gets 50%
```

**Breakpoint behavior:**
- `xl` (≥1280px): Side-by-side grid. Left ~66%, right ~34%. Right inner grid: `grid-rows-[1fr_1fr]`. Each card fills exactly 50% of the right column height.
- `lg–xl`: Same side-by-side layout.
- `sm–lg` (640–1024px): Right column `flex-row` (existing `sm:flex-row`). Each card 50% width, height auto.
- `<640px`: Stack vertically: carousel full width → card 1 full width → card 2 full width. Each card: `min-h-[160px]`.

**Implementation note:** Replace current `flex flex-wrap gap-5` wrapper with `grid`. Use `grid-template-columns: 1fr` on mobile, `minmax-based` two-column on xl. On right column inner container: switch from `flex flex-col sm:flex-row xl:flex-col` to `grid grid-rows-2 h-full` at xl breakpoint.

**Height control:** Each right card needs `h-full` to expand to fill its grid cell row. Outer right column must be `h-full` to match left. Grid intrinsic sizing computes automatically.

**Key constraint:** Do NOT set `min-height: 50vh` or any fixed vh value. Grid intrinsic sizing handles this.

## Hover System Refactor Plan

### Unified Hover Design Specification

**Chosen direction:** The "right-slide" animation pattern in `BestSeller/SingleItem.tsx` is visually premium. Extend globally.

**Unified hover overlay spec:**
- Trigger: CSS `group-hover` on parent div
- Animation: `translate-x-full → translate-x-0` (slides in from right)
- Position: `absolute right-0 top-0 h-full`
- Width: auto (content-driven, ~40–48px)
- Background: transparent (buttons have individual bg)
- Icons: outlined (eye, heart, cart) — square buttons
- Button size: `w-10 h-10` (40×40px)
- Button style: `bg-white border-l border-t border-[#E8E4DF] text-dark hover:bg-dark hover:text-white`
- Transition: `ease-linear duration-200`
- **Scope:** Both `ProductItem.tsx` AND `BestSeller/SingleItem.tsx`

**Action items:**
1. Refactor `ProductItem.tsx` hover overlay from bottom-slide to right-slide
2. Remove `rounded-[5px] shadow-1` from eye/wishlist buttons in `ProductItem.tsx` — replace with square `rounded-none border` style

### Wishlist Icon Logic Plan

**Target behavior:**
- Not in wishlist → outlined heart (black stroke, no fill)
- In wishlist → solid filled heart (black fill, NOT red)
- Click = toggle (optimistic UI)

**State management strategy:**
- Redux wishlist slice already exists: `addItemToWishlist` dispatched on add
- Add `removeItemFromWishlist` action to the wishlist slice
- Read wishlist state in each ProductItem via `useSelector`
- Check: `state.wishlist.items.some(w => w.item_id === item.id)` for icon state
- No local `savedToWishlist` useState — use Redux as single source of truth

**Backend sync plan:**
- `addToWishlist(item_id, "product")` → already exists in `lib/queries/wishlist.ts`
- Add `removeFromWishlist(item_id)` function to `lib/queries/wishlist.ts`
- Toggle flow: if in wishlist → call `removeFromWishlist`, dispatch `removeItemFromWishlist`
- Toggle flow: if not in wishlist → call `addToWishlist`, dispatch `addItemToWishlist`
- Optimistic UI: update Redux FIRST, then call Supabase. On failure, revert.

**Auth guard:** If `!user`, clicking wishlist → redirect to `/auth/signin` or show toast. Do not silently skip.

```

├── page.tsx                    (Server Component — fetch product)
```

**Page sections:**

   - Color selector (if `product.colors` array not empty)
4. **Action buttons:**
   - Wishlist toggle (heart icon — outline/filled per unified spec)
   - "Shop Now" CTA → `tracking.trackAffiliateClick()` → `window.open(item.affiliateLink, '_blank')`
   - Share button → Web Share API with copy-link fallback
   - "Add to Lookbook" (disabled placeholder, future feature)
5. **Product meta:** Commission rate, affiliate program name, deal tag
6. **View tracking:** `tracking.trackProductView(item.id, user?.id, sessionId)` on mount

### Related Products Section

**Relevance scoring (priority order):**
1. Same `category_id` (highest priority)
2. Shared `tags[]` overlap (≥1 matching tag)
3. Same `styles[]` overlap
4. Similar `colors[]` overlap
5. `is_trending = true` weight boost

**Backend query plan:**
```
GET /api/v1/products/related?productId={id}&limit=8
```
Query logic:
- Fetch current product's `category_id`, `tags`, `styles`, `colors`
- FROM `products` WHERE `id ≠ current` AND `is_hidden = false`
- ORDER BY: category match + tag overlap + same style + trending
- LIMIT 8
- MVP option: `WHERE category_id = $1 AND is_hidden = false ORDER BY is_trending DESC LIMIT 8`

**API response:**
```json
{ "success": true, "data": { "product": {...}, "related": [...] } }
```

**Responsive layout:**
- Mobile: Image full width → info below → related products horizontal scroll
- Tablet: Image 55% left, info 45% right, related 2-column grid
- Desktop: Image 60% left, info 40% right, related 4-column grid

## 3D Section Update Plan

### Background Gradient Redesign

**Current gradient (desktop):**
```css
linear-gradient(to right, #0A0A0A 0%, #1A1A1A 28%, #888880 42%, #F0EDE8 52%, #FFFFFF 100%)
```
**Problem:** `#888880` → `#F0EDE8` → `#FFFFFF` creates a muddy warm streak that clashes with the dark left panel and 3D model neutral lighting.

**Proposed gradient (desktop):**
```css
linear-gradient(to right, #0A0A0A 0%, #111111 30%, #1C1C1C 45%, #2A2A2A 60%, #0A0A0A 100%)
```

**Proposed gradient (mobile):**
```css
linear-gradient(to bottom, #0A0A0A 0%, #111111 50%, #161616 100%)
```

**Rationale:** Keep the entire section dark. The 3D models provide all visual interest. Background should recede, not compete.

**Performance plan:**
- Background is pure CSS — zero additional GPU cost
- Do NOT add `backdrop-filter: blur` to background — GPU-intensive on mobile
- Gradient rendered by browser compositor, not WebGL canvas — no 3D interaction

**Layer stacking (no change):**
1. Section background (CSS gradient)
2. 3D Canvas (WebGL, compositor layer)
3. Text overlay (z-10, above canvas)

## Responsiveness Strategy

### Global Breakpoints Applied

| Breakpoint | Width | Strategy |
|-----------|-------|----------|
| xs | 0–639px | Mobile-first base. Single column. |
| sm | 640–767px | 2-column where applicable. Horizontal scroll for products. |
| md | 768–1023px | 2–3 columns. Side-by-side hero. |
| lg | 1024–1279px | 3–4 columns. Full desktop layout. |
| xl | 1280–1535px | Max-width 1170px container. |
| 2xl | ≥1536px | Content capped at 1170px, background bleeds. |

### Per-Section Mobile Layout Strategy

**Hero:**
- Mobile (`<640px`): Single column. Carousel → AI Styling card → Lookbook card. HeroFeature below: 1-col.
- Tablet: Carousel + right cards side by side (`sm:flex-row` pattern).
- Desktop: Full CSS grid side-by-side, equal-height right cards.

**Categories:**
- Swiper at `slidesPerView: 2` from 0px — already correct.
- Navigation arrows: hide on mobile.

**New Arrivals, Best Sellers, Trending:**
- Mobile (`<640px`): Horizontal Swiper/scroll.
  - `slidesPerView: 1.2` (shows partial next card = scroll affordance)
  - `scroll-snap-type: x mandatory` on container
  - `scroll-snap-align: start` on each card
  - Hide scrollbar via CSS
- Tablet (`640–1023px`): 2-column grid. Vertical scroll acceptable.
- Desktop (`≥1024px`): 3–4 column grid.

**ThreeDSection:**
- Gradient update as specified. Responsive canvas height unchanged.

**Lookbook Preview:**
- Mobile: 1-column. Cards: `aspect-[3/4]`.
- Tablet: 2-column.
- Desktop: 4-column.

**Newsletter:**
- Mobile: Full-width. Input stacked above button.
- Desktop: Side-by-side (use existing `flex-col lg:flex-row` pattern).

### Navbar Spacing
- Hero section `pt-10 lg:pt-12` must account for fixed navbar height.
- Minimum top padding: navbar height + 16px.

### Touch-Friendly Spacing
- Minimum tap target: `44×44px` on all interactive elements.
- `w-10 h-10` (40px) buttons: add 4px responsive padding to reach 44px.
- Category pills: `py-2.5 px-4` minimum.

### Font Scaling Logic
- Hero H1: `text-4xl sm:text-5xl lg:text-5xl` — already scaled. Keep.
- Section headers: `text-3xl xl:text-4xl` — already scaled. Keep.
- Body copy: `text-sm` (14px) — readable at all breakpoints.
- Micro labels: `text-[10px]` — acceptable for eyebrow only.

### Image Scaling Strategy
- All product images: `next/image` with `sizes` prop: `(max-width: 640px) 75vw, (max-width: 1024px) 50vw, 25vw`.
- Hero carousel: `fill` + `object-cover` — already correct.
- Lookbook cards: `fill` + `object-cover` within `aspect-ratio` container.
- Enforce `next/image` everywhere — no raw `<img>` tags.

### Avoiding Layout Shift (CLS)
- Hero carousel skeleton: `min-h-[450px]` placeholder (existing L58 `HeroCarousel.tsx`) — keep.
- Lookbook section: Replace `return null` with skeleton cards (4 shimmer placeholders).
- New Arrivals: skeleton grid already implemented — keep.
